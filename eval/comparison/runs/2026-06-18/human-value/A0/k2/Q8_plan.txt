## Plan: file uploads in Express

**1. Multipart parsing — `multer`**
Express doesn't parse `multipart/form-data` on its own. `multer` is the standard middleware. Add it as the body parser for upload routes only (not globally).

**2. Decide storage destination** *(the main decision)*
- **Local disk** (`multer.diskStorage`) — simplest, fine for a single server / dev. Files land in e.g. `uploads/`.
- **Cloud (S3/R2/GCS)** via `multer-s3` or streaming — needed if you have multiple instances, serverless, or don't want files on the app box.

Default recommendation: start on local disk behind an abstraction, so swapping to S3 later is a one-file change.

**3. The upload route**
- `POST /upload` (or wherever it belongs) with `upload.single('file')` / `.array('files', n)`.
- Put it **behind your existing auth middleware** — uploads should almost never be anonymous.

**4. Validation & limits** *(security-critical)*
- `limits: { fileSize, files }` to cap size and count.
- `fileFilter` to allow-list MIME types **and** extensions (don't trust the client's `Content-Type` alone).
- Generate random server-side filenames (e.g. uuid) — never use the client filename in the path (prevents path traversal / overwrites).

**5. Persist metadata**
Store `{ id, originalName, storedPath/key, size, mimetype, ownerId, createdAt }` in your DB so files are queryable and ownership is enforced.

**6. Serving files back**
- Local: a guarded download route (check ownership) rather than blindly `express.static` on the upload dir.
- S3: return time-limited signed URLs.

**7. Error handling**
A small error middleware to turn multer errors (`LIMIT_FILE_SIZE`, etc.) into clean 4xx responses.

**8. Verify**
`curl -F file=@somefile ...` plus a test for the size/type rejection paths.
