## The one decision that shapes everything: in-app vs. push

| | **In-app real-time** (recommended first) | **Browser/OS push** |
|---|---|---|
| What it does | Bell/toast/badge updates **while the tab is open** | Reaches user **even when tab is closed** |
| Needs | A live connection (SSE / WebSocket / Supabase Realtime) | Service worker + Web Push API + VAPID keys |
| Gotchas | Reconnect logic, auth on the socket | iOS Safari is restrictive; permission prompts hurt opt-in |
| Effort | ~1вҖ“2 days | ~3вҖ“5 days on top of in-app |

**My recommendation: ship in-app real-time first, add push later as phase 2.** Most "live notification" value lands with in-app, and push is strictly additive.

## Phased plan

1. **Data model** вҖ” a `notifications` table: `id, recipient_user_id, type, payload jsonb, read_at, created_at`. This is the backbone regardless of transport.
2. **Write path** вҖ” your triggers (DB change / another user's action / backend job) insert a row. With Supabase, a DB trigger or your existing write code does it; otherwise an `emitNotification(userId, type, payload)` helper.
3. **Live transport** вҖ” Supabase Realtime subscription filtered by `recipient_user_id` (or an SSE endpoint `/notifications/stream` scoped to the authed user).
4. **Client UI** вҖ” a bell with unread count, a dropdown list, toast on arrival; mark-as-read writes `read_at`.
5. **Hardening** вҖ” auth on the channel, reconnect/backfill on reconnect (fetch unread since last seen), pagination, rate limiting.
6. **(Phase 2) Web Push** вҖ” service worker + VAPID for closed-tab delivery, reusing the same `notifications` table.
