Check `pg_stat_activity` first — it tells you in one query whether you're leaking connections, holding them too long, or just undersized. Run this while under load:

```sql
SELECT state, wait_event_type, count(*)
FROM pg_stat_activity
WHERE datname = current_database()
GROUP BY state, wait_event_type
ORDER BY count(*) DESC;
```

What the result points to, in the order I'd check:

**1. Lots of `idle in transaction` → connection leak (most common cause).** A code path acquires a connection, opens a transaction, and never commits/rolls back — usually a missing `finally`/`using` release on an error path, or a transaction left open across an `await` that throws. These connections never return to the pool, so it bleeds dry under sustained load. Find the worst offenders:

```sql
SELECT pid, state, now() - state_change AS idle_for, left(query, 80)
FROM pg_stat_activity
WHERE state = 'idle in transaction'
ORDER BY idle_for DESC;
```

If you see ages of minutes, that's the bug. Fix the release path; as a safety net set `idle_in_transaction_session_timeout`.

**2. Lots of `active` with long durations → slow queries holding connections.** The pool isn't leaking, it's just that each connection is tied up too long, so they all get checked out at once. Look at the longest-running ones (`now() - query_start`), then fix with indexes / query changes rather than a bigger pool.

**3. Counts look healthy but you still exhaust → pool simply undersized, or sized wrong relative to Postgres.** Sum the `max` pool size across *all* app instances and compare to Postgres `max_connections` (`SHOW max_connections;`). If total pool capacity exceeds `max_connections`, you exhaust at the server, not the pool. If it's well under and you're still starving, the pool's too small for your concurrency — but only raise it after ruling out 1 and 2, since bumping the pool to paper over a leak just delays the crash.

The reason to start with `pg_stat_activity` rather than your app's pool metrics: it shows what connections are *actually doing* on the server, which immediately separates "leaked," "slow," and "too few" — three problems with completely different fixes.
