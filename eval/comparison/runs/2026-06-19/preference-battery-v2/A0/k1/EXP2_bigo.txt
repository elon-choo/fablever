# Big-O Notation

Big-O describes **how an algorithm's cost grows as the input grows**. It throws away constants and small terms and keeps only the dominant factor — because what matters is the shape of the curve when `n` gets large, not whether something takes 3ms or 5ms on a tiny input.

The mental model: *"If I 10x the input, what happens to the work?"*

---

## The common ones, from best to worst

### O(1) — constant
Work doesn't depend on input size. Same cost for 10 items or 10 million.

```python
users[0]              # array index
cache[key]            # dict/hashmap lookup
len(my_list)          # Python tracks length, doesn't count
stack.append(x)       # push to end of list
```
**Where you hit it:** hash map lookups are the big one. This is *why* you reach for a `dict`/`Set` instead of scanning a list.

### O(log n) — logarithmic
Each step throws away half the remaining work. Doubling the input adds just *one* more step.

```python
# Binary search in a SORTED list
bisect.bisect_left(sorted_prices, target)
```
**Where you hit it:** binary search, balanced tree operations, and database **B-tree index lookups**. When a query is fast because there's an index on the column — that's log n instead of n.

### O(n) — linear
Look at each item once.

```python
total = sum(transactions)            # one pass
if "admin" in users_list:            # scans list until found  ← note: list, not set
max_price = max(products)
for line in open("file.txt"): ...    # reading a file
```
**Where you hit it:** any single loop over your data. Usually fine and often unavoidable — you have to read the data to process it.

### O(n log n) — "good sorting"
Linear work done log-n times. **This is the practical floor for general-purpose sorting.**

```python
sorted(orders, key=lambda o: o.date)   # Python's Timsort
df.sort_values("revenue")              # pandas
```
**Where you hit it:** every `.sort()` you call. Good news — if your algorithm is dominated by a sort, you're in fine shape.

### O(n²) — quadratic ⚠️
For each item, do work involving every other item. **This is the classic "it worked in dev, died in prod" bug.** A nested loop over the same data.

```python
# Find duplicate emails — the SLOW way
for i in range(len(users)):
    for j in range(len(users)):
        if i != j and users[i].email == users[j].email:
            ...
```
100 users → 10,000 comparisons. 10,000 users → **100,000,000**. The fix is almost always a hash set:

```python
# O(n) version
seen = set()
for u in users:
    if u.email in seen: ...        # O(1) membership check
    seen.add(u.email)
```

A sneakier version — the **accidental O(n²)** hiding behind clean-looking code:

```python
result = []
for item in items:
    if item not in result:   # `in` on a LIST is O(n) → whole thing is O(n²)
        result.append(item)
```

### O(2ⁿ) / O(n!) — exponential / factorial 💀
Cost roughly doubles (or worse) with each added element. Only viable for tiny `n`.

```python
# Naive recursive Fibonacci — recomputes everything
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)
```
**Where you hit it:** brute-forcing all subsets/permutations, naive recursion without memoization, the traveling-salesman done exhaustively. `fib(50)` will hang your machine; adding a cache makes it O(n).

## What it feels like in practice

Concrete operation counts make the curves visceral:

| n | O(1) | O(log n) | O(n) | O(n log n) | O(n²) |
|------|------|----------|---------|------------|----------------|
| 10 | 1 | 3 | 10 | 33 | 100 |
| 1,000 | 1 | 10 | 1,000 | 10,000 | 1,000,000 |
| 1,000,000 | 1 | 20 | 1,000,000 | 20,000,000 | **1,000,000,000,000** |

That last cell is why O(n²) silently passes tests on 100 rows and falls over on a real dataset.

---

## Things people get wrong

- **It's about growth, not speed.** An O(n²) algorithm can beat an O(n) one on small inputs (smaller constants). Big-O only tells you who wins *eventually*.
- **Constants are dropped on purpose.** Two loops back-to-back is `O(2n)` = **O(n)**. Big-O is about the curve's shape, not the exact count.
- **Worst case vs. average.** Hash lookups are O(1) *average* but O(n) worst case (all collisions). Quicksort is O(n log n) average, O(n²) worst. The headline number is usually the average/typical case.
- **Watch the hidden costs in "innocent" calls.** `x in a_list` is O(n). `x in a_set` is O(1). Same syntax, wildly different scaling — this is the single most common real-world trap.

---

## The practical takeaway

When something is slow at scale, scan for these two patterns first:
1. **A nested loop over the same data** → can it become a hash map? (n² → n)
2. **A membership test (`in`) against a list inside a loop** → make it a `set`. (n² → n)

Those two fixes cover the majority of "why is this so slow" moments you'll actually run into.
