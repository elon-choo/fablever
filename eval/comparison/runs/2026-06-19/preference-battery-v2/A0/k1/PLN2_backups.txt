## The plan, by store

**1. Aurora PostgreSQL — crown jewels (revenue/users/UTM). Priority 1.**
Aurora always has automated backups, so the real gaps are almost always these three:
- **Raise retention** to 14–35 days (`modify-db-cluster --backup-retention-period 35`, applies online).
- **Enable PITR confidence** — automated backups already give point-in-time restore; verify `LatestRestorableTime` is current (lag = broken).
- **Offsite + logical layer:** a nightly `pg_dump` (custom format, `-Fc`) to an S3 bucket in a *different region/account*, lifecycle to Glacier. This protects against the case automated snapshots don't: account compromise, accidental `DROP`, or AWS-side region loss. Cross-region automated snapshot copy is the managed alternative.
- **The thing everyone skips:** schedule a **quarterly restore drill** — restore the latest snapshot to a throwaway cluster, run a row-count/checksum sanity check, tear it down. Automate it so it's a cron job, not a memory.

**2. Supabase (ads). Priority 2 — most likely the *actual* zero-backup store.**
- If on Free tier: either upgrade to Pro (daily automated backups) or, cheaper, run a **scheduled `pg_dump` to object storage** (GitHub Action / cron / Supabase Edge cron) — same `-Fc` dump pattern, encrypted, offsite.
- If you need tight RPO, add the **PITR add-on**.
- Same restore-drill requirement applies.

**3. Neo4j knowledge graph. Priority 3 (or skip).**
- It's derived from source code, so DR = re-run `index_project`. If re-indexing is cheap, document that as the recovery procedure and don't bother backing it up. If re-indexing is expensive/slow, schedule `neo4j-admin database dump` to the same offsite bucket.

**4. PostHog** — SaaS, they own backups. No action beyond noting it in the DR doc.

## Make it a guarantee, not a one-off
1. **Automate** every dump/snapshot (cron/scheduled job, `caffeinate`-wrapped if local per your long-job conventions).
2. **Monitor freshness** — alert if the newest backup object is older than its RPO. A silent-failing backup job is worse than none (false confidence).
3. **Restore drills on a schedule** — this is the evidence step; until a restore succeeds, treat the store as *unbacked* in your status reporting.
4. **One-page DR runbook** — per store: where backups live, exact restore command, owner, last-tested date, measured RPO/RTO.
