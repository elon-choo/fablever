It's an infinite loop. 🐛

The problem is the strict equality `x !== 1` combined with floating-point accumulation. `0.1` isn't exactly representable in IEEE 754 double precision, so the running sum drifts and **never lands exactly on `1.0`**:

```
0
0.1
0.2
0.30000000000000004
0.4
0.5
0.6
0.7
0.7999999999999999
0.8999999999999999
0.9999999999999999   ← the "1" step, but it's not === 1
1.0999999999999999   ← overshot; condition still true
...
```

The loop sails right past 1 and runs forever (well, until `x` grows so large that `x += 0.1` stops changing it around 2^53, at which point it's *truly* stuck).

**Fixes, roughly in order of preference:**

1. **Iterate over integers, scale inside:**
   ```js
   for (let i = 0; i < 10; i++) { tick(i * 0.1); }
   ```
   Cleanest — the loop variable is exact, no drift in the *condition*. (`i * 0.1` still has FP rounding, but the count is correct.)

2. **Use a `<` bound instead of `!==`** if you keep the float accumulator:
   ```js
   for (let x = 0; x < 1; x += 0.1) { tick(x); }
   ```
   Terminates — but watch out, this runs **10 times** (last `x ≈ 0.9999...`), and a tiny upward drift could occasionally drop the final iteration. Integer counting avoids that ambiguity.

3. **Epsilon comparison** only if you genuinely need "approximately 1":
   ```js
   for (let x = 0; Math.abs(x - 1) > 1e-9; x += 0.1)
   ```
   Works but is the most fragile/ugly option.

The general rule: **never use `===`/`!==` as a loop terminator on an accumulated float.** Drive loops with integers and derive the float each iteration.
