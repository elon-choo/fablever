## Idempotency in APIs

**Definition:** An operation is *idempotent* if making the same request multiple times has the same effect on the server's state as making it once. The client can safely retry without causing additional or unintended changes.

Note the subtlety: idempotency is about the **resulting state**, not necessarily the **response**. A second `DELETE` might return `404` instead of `200`, but the state ("the resource is gone") is unchanged — that's still idempotent.

## Idempotency by HTTP method

| Method | Idempotent? | Why |
|--------|-------------|-----|
| `GET` | ✅ Yes | Read-only; never changes state |
| `PUT` | ✅ Yes | Replaces the resource with a full representation — repeating sets the same value |
| `DELETE` | ✅ Yes | Resource ends up deleted regardless of repeat count |
| `POST` | ❌ No (by default) | Typically *creates* a new resource each call |
| `PATCH` | ⚠️ Depends | `{"status": "paid"}` is idempotent; `{"balance": "+10"}` is not |

This is a *contract* the HTTP spec asks you to honor, not something the protocol enforces. You have to implement your handlers so they actually behave this way.

## Why it matters

1. **Networks are unreliable.** A client sends a request, the server processes it, but the response is lost in transit. The client doesn't know if it succeeded, so it retries. Without idempotency, a retried `POST /payments` could charge the customer **twice**.

2. **Safe automatic retries.** Load balancers, client SDKs, and message queues retry failed requests by default. Idempotency is what makes those retries safe rather than dangerous.

3. **At-least-once delivery → effectively-once.** Distributed systems usually guarantee *at-least-once* delivery (duplicates happen). Idempotency turns that into *effectively-once* processing.

## Making non-idempotent operations idempotent

The classic pattern (used by Stripe, PayPal, etc.) is an **idempotency key**:

```http
POST /v1/charges
Idempotency-Key: a1b2c3-unique-per-operation
```

How the server handles it:
- First time it sees the key → process the request, **store the key + the response**.
- Sees the same key again → **skip reprocessing**, return the stored response.

Implementation considerations:
- The key should be generated by the **client**, unique per logical operation (often a UUID).
- Store keys with a **TTL** (e.g., 24h) so they don't grow forever.
- Watch for **race conditions** — two concurrent requests with the same key. Use a unique DB constraint or a lock so only one wins.
- Decide what to do on a **retry of a still-in-progress** request (usually return `409 Conflict` or block until done).

## A useful mental model

> *Idempotent = "you can press the button as many times as you want; the outcome is the same as pressing it once."*

An elevator call button is idempotent (the elevator still comes once). A "submit payment" button generally is **not** — which is exactly why well-built checkout flows disable the button and use idempotency keys behind the scenes.
