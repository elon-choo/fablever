## 🔴 Critical

**1. Hardcoded live secret in source**
```js
const API_KEY = 'sk_live_abc123';
```
The `sk_live_` prefix is a **production** secret key (Stripe-style convention). Hardcoding it means:
- It gets committed to version control — and stays in git history even after you "remove" it later.
- Anyone with repo read access has your live key.
- **Remediation:** Move it to an environment variable / secret manager, and rotate this key now (assume it's burned the moment it's in a commit). `const API_KEY = process.env.API_KEY;`

**2. `fetch` strongly implies this is client-side → the key is fully public**
`fetch` running in a browser ships the key in the JS bundle and exposes it in the Network tab to every visitor. A **live secret key must never reach the client.** Calls that need it must be proxied through your backend; the browser talks to your server, your server holds the secret.
- If this is server-side (Node), it's less catastrophic but issue #1 still stands.

## 🟡 Worth fixing

**3. Auth header format is likely wrong**
```js
headers: { Authorization: API_KEY }
```
Most APIs expect a scheme prefix: `Authorization: \`Bearer ${API_KEY}\`` (or `Basic <base64>`). Passing the raw key often fails auth or behaves unexpectedly — confirm against the API's docs.

**4. `url` is unvalidated**
The key is sent to whatever `url` holds. If `url` is ever dynamic/attacker-influenced, you've handed your secret to an arbitrary host. Pin it to a known constant/allowlist.

**5. No error handling**
No `.catch` / response-status check — failures (including auth failures that might leak info in the response) pass silently.

## Suggested shape (server-side)
```js
const API_KEY = process.env.API_KEY;
if (!API_KEY) throw new Error('API_KEY not set');

const res = await fetch(url, {
  headers: { Authorization: `Bearer ${API_KEY}` },
});
if (!res.ok) throw new Error(`Request failed: ${res.status}`);
```
