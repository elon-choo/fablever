Two real problems here, one serious.

**1. Hardcoded live secret (critical).** `sk_live_abc123` is a production key sitting in source. The `sk_live_` prefix means it's a real, live credential — anyone with repo access (or anyone it's ever been pushed to) has it. Move it to an environment variable / secret manager and read it at runtime:

```js
const API_KEY = process.env.API_KEY;
```

If this key was ever committed, treat it as compromised and **rotate it** — removing the line doesn't scrub it from git history.

**2. Malformed Authorization header (will likely break the request).** Most APIs expect a scheme prefix — `Bearer`, `Basic`, etc. Sending the bare key as the header value usually gets rejected as `401`. You almost certainly want:

```js
fetch(url, { headers: { Authorization: `Bearer ${API_KEY}` } });
```

Confirm the scheme against the API's docs (some use `Basic`, a few want the raw token, but bare-value is the exception, not the default).

Minor: there's no check that `fetch` succeeded — for production code you'd want to handle a non-`ok` response, but that's secondary to the two above.
