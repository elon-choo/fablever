Idempotency means an API operation produces the same result no matter how many times you call it with the same input. One call or ten identical calls leave the system in the same final state, and ideally return the same response. The classic example: `DELETE /users/123` deletes the user on the first call; the next four calls find nothing left to delete and report the resource is already gone. State after one call equals state after five.

**What's idempotent by default.** In HTTP terms, `GET`, `PUT`, and `DELETE` are expected to be idempotent. `GET` just reads. `PUT` replaces a resource with the exact payload you send, so resending it overwrites with the same values. `DELETE` removes a thing that's then already removed. The notable exception is `POST`, which usually *creates* something new each time — call "create order" twice and you get two orders. That's the case people have to engineer around.

**Why it matters.** Networks are unreliable, and the dangerous failure isn't the request that clearly errors — it's the one where the server processed your request but the response got lost on the way back. The client has no way to tell "it didn't happen" from "it happened but I didn't hear about it." So the client retries. If the operation isn't idempotent, that retry charges the card twice, sends two emails, or places a duplicate order. Idempotency makes retries safe, which is what lets clients, load balancers, and message queues retry automatically without coordinating with you.

**How non-idempotent operations are made safe.** For payment-like `POST` operations, the standard pattern is an **idempotency key**: the client generates a unique key (often a UUID) and sends it with the request, usually in a header. The server records the key with the result of the first request. If a request arrives with a key it has already seen, the server skips the work and returns the stored result instead of doing it again. This is how Stripe, for instance, guarantees a retried charge doesn't bill the customer twice. The key is typically scoped to a time window and to the specific request, so reusing it with a *different* body is treated as an error rather than silently mismatched.

Two distinctions worth keeping straight: idempotency is about the *resulting state*, not necessarily an identical response body — a second `DELETE` might legitimately return `404` instead of `204`, yet the operation is still idempotent because the state is unchanged. And idempotency is not the same as *safety*: a "safe" method (like `GET`) changes no state at all, while an idempotent method may change state once but never compounds it on repeat.

The practical payoff is that idempotency is what makes a distributed system tolerant of the retries it can't avoid — without it, every dropped response is a potential double-charge or duplicate record.
