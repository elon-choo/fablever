## The actual gap

You're not at zero — you have **PostHog** (product analytics: events, funnels, CTAs) and a homegrown **session-log/recall** system. What's missing is the *operational* layer, which is what "no monitoring" usually means:

- **Error tracking** — you find out prod broke when a user tells you.
- **Uptime / availability** — nothing pings the service or alerts on down.
- **Latency & throughput** — no view of slow requests or DB pressure.
- **Job/batch health** — your long-running render/sim/backfill jobs and crons have `ops-monitor` + `jobreg`, but no durable alerting when one dies silently.
- **Log aggregation** — logs are scattered across PM2/launchd, not searchable in one place.

## Recommended approach

**Managed SaaS first, free tiers, expand later.** You already run a lot of infra (PM2/launchd bridge, docbroker, agent fleet). Self-hosting Prometheus/Grafana/Loki adds another thing *you* have to keep alive — the wrong move when the goal is to stop being surprised. Go managed for the win, revisit self-hosting only if cost or data-residency forces it.

## Phased plan

**Phase 0 — stop flying blind (highest ROI, ~a day):**
1. **Sentry** on the prod 사주/천기문 web app + backend. Frontend + server SDK, release tagging, source maps. This alone catches the majority of "user hit an error and we never knew."
2. **Uptime + cron dead-man's-switch** via Healthchecks.io or Better Stack: external ping on the public service, and a heartbeat from every cron + long job. This directly closes the "silent stuck job" gap `ops-monitor` watches for in-session but loses once the session ends.
3. **One alert channel** — route Sentry + Healthchecks to a single Slack/Telegram/email so alerts actually reach you.
