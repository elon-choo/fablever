The critical problem: a live secret key is hardcoded in source. Everything else is secondary.

**1. Hardcoded live credential (critical).** `sk_live_abc123` is a production secret key sitting in source. It will land in version control, build artifacts, and anyone's clone. Move it to an environment variable / secret manager and read it at runtime:

```js
const API_KEY = process.env.API_KEY;
```

If this key is real and has ever been committed, rotate/revoke it now — removing the line doesn't undo the exposure, since it stays in git history.

**2. If this `fetch` runs in the browser, it's worse.** A `sk_live_` key is a server-side secret. In frontend code it's shipped to every visitor and fully readable in devtools — no env var fixes that. Calls using a live secret key must go through your backend; the browser should never hold it.

**3. Authorization header likely needs a scheme prefix.** Most APIs (Stripe, and Bearer-token APIs generally) expect `Authorization: Bearer <key>`, not the bare key:

```js
fetch(url, { headers: { Authorization: `Bearer ${API_KEY}` } });
```

Confirm against the specific API's docs — a few use a raw key or a custom header, but bare-key in `Authorization` is the uncommon case.
