# 증거와 신뢰성 — fablever를 평가하는 법 (먼저 읽어주세요)

**이 파일은 검증 지도입니다 — 각 주장이 무엇인지, 무엇을 의도적으로 주장하지 *않는지*, 그리고
각각을 확인할 정확한 파일이 무엇인지를 담았습니다.** 신뢰하지 말고 검증하세요. fablever의 신뢰성은
형용사에 기대지 않습니다. 당신이 몇 분 안에 직접 확인할 수 있는 세 가지에 기댑니다:

1. **한정된(bounded)** 주장을 하고, 무엇을 주장하지 *않는지*를 명시한다.
2. **재현 가능한 평가**를 함께 싣고, **자신에게 불리하게 나온 결과까지** 공개한다.
3. **한계를 글로, 같은 저장소 안에, 시키지 않아도** 인정한다.

한계를 숨기는 도구는 한계를 나열하는 도구보다 덜 신뢰할 만합니다. 이 파일은 각 강점을 그것을
입증하는 산출물에 연결하고, 경계를 명확히 합니다. 아래 모든 행을 "우리 말을 믿어라"가 아니라
"이 파일을 읽고 스스로 판단하라"로 받아들이세요.

> **정직성 계약(binding, 저장소 전체).** 스캐폴딩은 기본 실력의 곱셈기이지 결코 대체재가 아니다.
> 이것은 **스타일(style)** 이식이지 능력(capability) 이식이 아니다. 개발자 생산성 크기에 대한 주장은
> 하지 않는다. 증거가 프로젝트에 불리하게 나온 경우에도 그대로 공개한다. 이 저장소의 어떤 문장이라도
> 과장한다면 그것은 버그다 — 이슈를 열어주면 인정·수정·반박된다.

## 1 · 이 프로젝트가 주장하지 **않는** 것 (가장 중요한 신뢰 신호)

대부분의 "당신의 모델을 \<X\>처럼 행동하게 만들기" 패키지는 최상급 표현으로 시작합니다. 이 프로젝트는
자신의 천장(한계)으로 시작합니다. 이 비(非)주장들은 핵심을 떠받치며 저장소 전체에 명시되어 있습니다:

- **생산성-크기 주장 없음.** "생산성을 N% 향상" 같은 것은 주장하지 *않는다*. 해당 명제(T2)는 명시적으로
  **강등**되어, 메커니즘 뒷받침이 있는 가설이며 크기는 미측정 상태다. 검증:
  [`whitepaper/05-consensus-and-claims.md`](whitepaper/05-consensus-and-claims.md),
  [`docs/PUBLICATION-READINESS.md`](docs/PUBLICATION-READINESS.md).
- **능력 업그레이드 아님.** 약한 모델의 추론 천장을 끌어올릴 수 없다. 작업 *방식*과 협업 *구조*를 바꾼다.
  검증: README의 "What this is and isn't" 박스와
  [`whitepaper/01-what-this-is.md`](whitepaper/01-what-this-is.md) §1.1.
- **"병렬이 단일을 이긴다"가 아님.** 자체 통제 A/B가 순진한 "에이전트 많을수록 이긴다" 서사를 *반증*한다.
  검증: [`whitepaper/03-results.md`](whitepaper/03-results.md) §3.2.
- **"비판으로부터 안전함"이 아님.** "어떤 유효한 비판도 없이 출판 가능"이라는 지향적 기준(T1)은 보장이
  아니라 **과정**으로 강등되었다. 검증: `docs/PUBLICATION-READINESS.md`.

약점을 미리 인정하는 프로젝트는 그러지 않는 프로젝트보다 구조적으로 과장을 적발하기 더
어렵습니다 — 그래서 비주장들이 이 파일 맨 위, 어떤 강점보다도 앞에 자리합니다.

## 2 · 근거 있는 강점 — 그리고 각각을 검증할 위치

| 강점 | 실제로 참인 것 | 검증 위치 |
|------|----------------|-----------|
| **주장이 아니라 측정됨** | 사전 등록·계층화·모델 교체·조건 블라인드 평가 하니스와 그 *결과*(공개된 **null/부정** 결과 포함)를 싣는다. | `eval/ab-harness.mjs`, `eval/results-2026-06-15.md`, `eval/results-2026-06-15-hard.md` |
| **평가에 통제군이 있음** | 구조를 렌즈-분류·드로-수 교란요인에서 분리하는 4개 arm(baseline, prompt-matched, draw-matched, panel). 이걸 통제하는 "에이전트" 도구는 드물다. | `eval/ab-harness.mjs`, `whitepaper/02-methodology.md` §2.2 |
| **최선 사례 결과, 강건하게 채점됨** | 최신 모델(GPT-5.5 + Gemini-3.1-pro-preview)로 비용 무제한 파이프라인이 심어둔 결함 **16/18**개를 **5개 크로스모델 심판단(4 GPT + 1 Gemini)** 다수결에서 탐지, **정밀도 0.74(모든 구성 중 최고)**; 구모델 run 피크는 **18/18** 재현율. 각 수치는 그것을 만든 모델로 라벨되며, 스크립트 + 원시 데이터가 함께 커밋되어 있다. | `whitepaper/03-results.md` §3.3, `eval/ultra/` |
| **오케스트레이션은 산문이 아니라 실행 코드** | 실제 `parallel()` 배리어, 스키마 강제 출력, JS가 소유한 게이트 — "X처럼 행동하라" 텍스트가 아니라 런타임 테스트가 있다. | `orchestration/recipes/*.mjs`, `test/orchestration-runtime-test.js` |
| **크로스모델 비상관화** | 기본 OFF인 arm이 가중치가 다른 리뷰어(GPT/Gemini)를 더해, 같은 계열 패널이 구조적으로 못 보는 부류를 잡는다. 꺼두면 오버헤드 0. | `fusion/fusion-server.js`, `whitepaper/01-what-this-is.md` §1.4 |
| **의존성 0, 공급망 청결** | npm 의존성 없음, `postinstall`/`prepare` 스크립트 없음. 유일한 네트워크/키 표면은 옵트인 서버 하나. | `package.json`, `fusion/fusion-server.js` |
| **페일세이프, 되돌릴 수 있음** | 훅은 오류 시 열린 채 실패; `FABLE_PROFILE=off`로 비활성화; `install.sh --uninstall`이 이전 설정을 결정적으로 복원. | `install.sh`, `claude-code/hooks/fable-subagent.js`, `claude-code/lib/mcp-remove.js` |
| **적대적으로 검토됨, 기록으로 남음** | 3라운드 × (Claude 전문가 페르소나 + GPT + Gemini) 합의, **실패한 공격까지 함께 기록**. | `docs/PUBLICATION-READINESS.md` |
| **출처(provenance)** | Anthropic의 *공개된* Fable 프롬프팅 가이드에서 증류(역공학 아님, 유출 콘텐츠 아님). | `docs/RESEARCH.md`, `NOTICE` |
| **투명한 실험 기록** | 날짜별 실험 로그 전체를 공개 — 실패하거나 **프로젝트에 불리하게 나온 실행**(포화 픽스처에서의 패배, 역효과를 낸 에스컬레이션)까지 포함. | [`whitepaper/08-experiment-log.md`](whitepaper/08-experiment-log.md) |

## 3 · "Fable처럼 행동하라" 래퍼들의 흔한 실패 모드를 어떻게 보완하는가

이것은 fablever를 Fable 스타일 프롬프트 패키지의 **전형적 패턴**과 *카테고리로서* 비교한 것입니다 —
이것들은 잘 알려진 실패 모드이지, 특정 명명된 프로젝트에 대한 감사가 **아닙니다**. 각 대비는 위의
산출물에 근거합니다. 읽고 판단하세요.

| 래퍼의 흔한 실패 모드 | fablever의 입장 | 근거 있음? |
|------------------------|------------------|------------|
| **측정 전무** ("그냥 잘 됨") | 자신을 *반증*할 수 있는 평가를 싣고, 돌렸고, 불리하게 나온 결과(쉬운 작업에서 패널=순수 비용)까지 공개. | ✅ `eval/` + 결과 문서 |
| **과장** ("생산성 10배") | 생산성 주장을 명시적으로 강등; "스타일이지 능력 아님"을 명시. | ✅ `whitepaper/05`, `docs/PUBLICATION-READINESS.md` |
| **산문 위약(prose placebo)** — 오케스트레이션을 바꾸지 못하는 "Fable처럼 해" 텍스트 | 스타일 층과 **실행되는** Workflow 제어흐름 층을 분리하고 런타임 테스트로 검증 — 그리고 A/B가 그 요인을 아직 완전히 분리하지 못함을 솔직히 명시. | ✅ `orchestration/`, `test/` (+ `whitepaper/02`에서 교란요인 인정) |
| **단일 계열 리뷰 맹점** | 가중치가 다른 모델의 선택적 크로스모델 검증. | ✅ `fusion/fusion-server.js` |
| **무거운 의존성 / 설치시 스크립트** (공급망 리스크) | 의존성 0; postinstall/prepare 없음. | ✅ `package.json` |
| **되돌릴 수 없는 / 불투명한 설치** | 토글 가능 + 설정 복원형 결정적 언인스톨. | ✅ `install.sh` |
| **인정된 한계 없음** | 평가자가 확장하도록 초대하는 상설 공개 한계 목록. | ✅ `whitepaper/06-limitations.md` |

관통하는 한 가지: **fablever는 더 큰 주장으로 차별화되지 않는다 — 자신의 카테고리 안에서, 반증
테스트를 만들고·돌리고·한계까지 공개한 유일한 도구라는 점으로 차별화된다.** 그것이 정밀 검토를
견디는 속성이다 — 위의 산출물에 대조해 확인하라.

## 4 · 우리가 인정하는 한계 (당신이 찾을 필요 없도록)

전체 목록: [`whitepaper/06-limitations.md`](whitepaper/06-limitations.md). 핵심만:

- 대표 결과(최신 모델 **16/18**, 구모델 피크 **18/18**)는 **작은(n=6), 저자가 심은** 픽스처에서
  **단일 생성 실행**으로 얻은 **결함 탐지** 결과다 — 강건하게 *채점*된 것이지 생산성 수치가 아니다.
  "최신"이 여기서 재현율을 올리지는 못했다.
- 정밀도(최신 **0.74**, 구모델 **0.63**)는 태스크당 3개 정답 키가 설정한 **하한**이지, 측정된 환각
  비율이 아니다(적대적 반박 패스가 사실상 아무것도 떨어뜨리지 못함 — 추가 발견들은 노이즈가 아니라 진짜 결함).
- 오케스트레이션의 재현율 이득은 "병렬 구조"가 아니라 **렌즈-분류 + 드로-수**이며; 패널의 구조적 승리는
  정밀도이고, 쉬운 작업에서는 순수 비용이다.
- **숨기지 않은 미해결 항목:** 아카이브된 출처 스냅샷(D2)과 토큰/벽시계 비용-방향은 미충족; 개발자 대면
  생산성 A/B는 아직 실행되지 않음.

여기 나열되지 않은 약점을 발견했다면, 그것은 기여다 — 이슈로 등록되어야 하고, 이 저장소 자체의 규칙에
따라 인정·수정·반박될 때까지 차단 요소가 된다.

## 5 · 전부 재현하기 (믿지 말고 — 돌려보라)

단계별 명령: [`whitepaper/07-reproduce.md`](whitepaper/07-reproduce.md).

- **오프라인, 키 불필요:** `node eval/ultra/score.mjs`는 대표 수치 뒤의 후보/확정(candidate/confirmed)
  카운트를 커밋된 원시 데이터([`eval/ultra/raw/`](eval/ultra/raw/))에서 곧바로 재계산한다.
- 테스트: `npm test` (오케스트레이션 계약 + 런타임 스모크 + MCP + fusion + 설치 라이프사이클).
- Claude 전용 A/B(`eval/ab-harness.mjs`)는 맨-`node` 스크립트가 아니라 **Workflow 도구 모듈**이다;
  그 기록된 출력은 `eval/results-2026-06-15*.md`에 커밋되어 있다.
- 크로스모델 ULTRA 파이프라인은 — 스크립트 **및** 그것이 산출한 원시 JSON까지 —
  [`eval/ultra/`](eval/ultra/)에 커밋되어 있다; 라이브 심판 단계만 당신의 API 키를 필요로 한다(돌리기
  전에 스크립트를 읽어볼 것 — 공급망 위생).

## 6 · 가장 빠른 신뢰성 체크 (시간 제한이 있는 검토자용)

1. 여기 §1을 읽어라 — 비(非)주장을 인정하는가? (그렇다)
2. `eval/results-2026-06-15.md`를 열어라 — 프로젝트에 **불리한** 결과를 보고하는가? (그렇다 — 포화된 픽스처에서 패널 = 5배 비용)
3. `whitepaper/06-limitations.md`를 열어라 — 한계 목록이 구체적이고 자기-불리한가? (그렇다)
4. `package.json`을 확인하라 — 의존성 수. (0)
5. `docs/PUBLICATION-READINESS.md`를 훑어라 — 실패한 공격이 성공한 수정과 함께 기록되어 있는가? (그렇다)

다섯 가지를 모두 통과하는 프로젝트는, 구성상, 과장으로 비난하기 어렵다. 이 문장이 아니라 다섯 가지
체크로 판단하라.
